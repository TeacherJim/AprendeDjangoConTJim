Tutorial 02

Entornos controlados + Django

Cada proyecto Django necesita sus propias dependencias aisladas del resto de tu computadora. En esta misión creas un entorno virtual, instalas Django 6.0 y aprendes a documentar lo que instalaste.

Inicial ~40 min 10 pasos

Resultado final

Al terminar este tutorial tendrás:

  • Una carpeta de proyecto llamada mi_primer_django con un entorno virtual .venv dentro.
  • Django 6.0.x instalado solo dentro de ese entorno, no en todo el sistema.
  • La capacidad de activar y desactivar el entorno virtual cuando trabajes.
  • Un archivo requirements.txt que lista las dependencias exactas del proyecto.

Concepto clave

Un entorno virtual es como darle a cada proyecto su propia mochila de herramientas. El Proyecto A puede llevar Django 6.0 y el Proyecto B otra versión, sin que se peleen en la misma computadora.

Sin entorno virtual: Computadora ├── Proyecto A → Django X └── Proyecto B → Django X (misma versión forzada) Con entornos virtuales: Computadora ├── Proyecto A │ └── .venv → dependencias de A └── Proyecto B └── .venv → dependencias de B

Antes de empezar

Confirma que completaste el Tutorial 01 o que cumples estos requisitos:

  • Python 3.12 o superior responde en la terminal (python --version o python3 --version).
  • VS Code está instalado con la extensión Python de Microsoft.
  • Sabes abrir una terminal integrada en VS Code.
  • Entiendes que los comandos de este tutorial se ejecutan en la terminal, no en el editor de texto.

Nota de Teacher Jim

El error más frecuente en este tutorial es instalar Django sin activar el entorno virtual. Si no ves (.venv) al inicio de tu línea de comandos, detente antes de instalar nada.

Mapa de la misión

Diez pasos en orden. Marca cada uno al completarlo.

  1. Crear la carpeta mi_primer_django.
  2. Crear el entorno virtual con python -m venv .venv.
  3. Activar el entorno (PowerShell, CMD o macOS/Linux).
  4. Verificar que el intérprete apunta al .venv.
  5. Actualizar pip.
  6. Instalar Django 6.0 con restricción de versión.
  7. Comprobar la versión con django --version.
  8. Revisar paquetes instalados con pip list.
  9. Generar requirements.txt con pip freeze.
  10. Desactivar el entorno y checkpoint final.
1

Crear la carpeta del proyecto

Qué vamos a hacer

Crear una carpeta dedicada donde vivirá tu primer proyecto Django y entrar en ella desde la terminal.

Por qué

Django no se instala "en el aire". Necesitas un directorio de trabajo donde colocarás el entorno virtual, el código y más adelante el proyecto. Separar carpetas evita mezclar archivos de distintos experimentos.

Código

Terminal — carpeta de proyectos

Terminal — Documentos, home del usuario u otra ubicación de tu elección

mkdir mi_primer_django
cd mi_primer_django

Dónde va

La carpeta puede estar donde prefieras. Lo importante es recordar la ruta. En VS Code: File → Open Folder y selecciona mi_primer_django.

Qué debería ocurrir

El prompt de la terminal muestra que estás dentro de mi_primer_django. La carpeta está vacía por ahora.

mi_primer_django/ └── (vacía)

Verifica

Comprueba

Ejecuta pwd en macOS/Linux o cd sin argumentos en Windows PowerShell. La ruta debe terminar en mi_primer_django.

2

Crear el entorno virtual .venv

Qué vamos a hacer

Generar un entorno virtual llamado .venv dentro de la carpeta del proyecto.

Por qué

El módulo venv de Python crea una copia aislada del intérprete y un espacio para instalar paquetes sin afectar el Python global de tu sistema ni otros proyectos.

Código

PowerShell — mi_primer_django 
python -m venv .venv

Si python no funciona, prueba py -m venv .venv.

Terminal — mi_primer_django 
python3 -m venv .venv

Dónde va

El comando se ejecuta dentro de mi_primer_django. El punto en .venv indica que la carpeta del entorno se crea aquí mismo (convención habitual).

Desglose del comando

python — invoca el intérprete.

-m venv — ejecuta el módulo estándar venv como programa.

.venv — nombre de la carpeta del entorno (puede ser otro nombre, pero .venv es la convención que usaremos en todo el curso).

Qué debería ocurrir

El comando termina sin errores. Aparece una carpeta .venv (puede estar oculta en el explorador de archivos; VS Code sí la muestra).

mi_primer_django/ └── .venv/ ├── Scripts/ (Windows) ├── bin/ (macOS/Linux) ├── Lib/ o lib/ └── pyvenv.cfg

Verifica

Comprueba

En VS Code, el explorador de archivos muestra la carpeta .venv. Si el comando falló, no continúes: revisa que Python esté instalado (Tutorial 01).

Error común

No module named venv — Tu instalación de Python no incluye venv. En Linux a veces necesitas instalar python3-venv desde el gestor de paquetes de tu distribución.

3

Activar el entorno virtual

Qué vamos a hacer

Activar el entorno virtual para que la terminal use el Python y los paquetes de .venv, no los globales.

Por qué

Crear el entorno no basta: hasta que no lo activas, pip install podría instalar paquetes en el Python del sistema. La activación cambia qué intérprete responde cuando escribes python.

Código

PowerShell — mi_primer_django 
.venv\Scripts\Activate.ps1
CMD — mi_primer_django 
.venv\Scripts\activate.bat
Terminal — mi_primer_django 
source .venv/bin/activate

Qué debería ocurrir

El inicio de tu línea de comandos muestra un prefijo como (.venv):

Salida esperada
(.venv) PS C:\Users\TuNombre\mi_primer_django>

Error común en PowerShell

Si ves: running scripts is disabled on this system o un mensaje sobre ExecutionPolicy, PowerShell está bloqueando el script de activación.

Opción prudente 1: Usa CMD en lugar de PowerShell y ejecuta .venv\Scripts\activate.bat.

Opción prudente 2: En PowerShell, para la sesión actual únicamente (no cambia la política global del sistema):

PowerShell — política de ejecución (solo esta sesión) 
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope Process
.venv\Scripts\Activate.ps1

Cuidado aquí

No recomiendo desactivar la seguridad de PowerShell de forma permanente en todo el sistema sin entender las consecuencias. Para aprender Django, activar el entorno en CMD o usar -Scope Process es suficiente.

Verifica

Comprueba

Debes ver (.venv) al inicio del prompt. Sin ese prefijo, no instales Django todavía.

4

Verificar el intérprete activo

Qué vamos a hacer

Confirmar que el comando python apunta al ejecutable dentro de .venv, no al Python global.

Por qué

Es la prueba definitiva de que el entorno está activo. Si la ruta no incluye .venv, cualquier pip install irá al lugar equivocado.

Código

PowerShell — con (.venv) activo 
where python
Terminal — con (.venv) activo 
which python

Qué debería ocurrir

La primera ruta listada debe contener .venv:

Ejemplo en Windows
C:\Users\TuNombre\mi_primer_django\.venv\Scripts\python.exe

Verifica

Comprueba

En VS Code, abre la paleta (Ctrl+Shift+P), busca Python: Select Interpreter y elige el que incluye .venv. Así el editor y la terminal coinciden.

Error común

La ruta apunta a AppData o Program Files sin pasar por .venv. El entorno no está activo. Vuelve al paso 3.

5

Actualizar pip

Qué vamos a hacer

Actualizar el gestor de paquetes pip dentro del entorno virtual antes de instalar Django.

Por qué

Una versión reciente de pip resuelve mejor las dependencias y evita errores al instalar paquetes grandes como Django. Solo actualizamos dentro del .venv activo.

Código

Terminal — (.venv) activo

Terminal — mi_primer_django, entorno virtual activado

python -m pip install --upgrade pip

Dónde va

Se ejecuta en la terminal con (.venv) visible. El flag -m pip garantiza que usas el pip del Python activo.

Qué debería ocurrir

Verás mensajes de descarga e instalación. Al final, algo como Successfully installed pip-24.x o un mensaje indicando que pip ya estaba actualizado.

Verifica

Comprueba

Ejecuta python -m pip --version. La ruta del pip debe incluir .venv.

6

Instalar Django 6.0

Qué vamos a hacer

Instalar Django restringiendo la versión a la rama 6.0 para mantener consistencia con el resto del curso.

Por qué

Sin restricción, pip podría instalar una versión distinta y los ejemplos del curso no coincidirían. Limitar a >=6.0,<6.1 asegura Django 6.0.x.

Código

Terminal — (.venv) activo 
python -m pip install "Django>=6.0,<6.1"

Dónde va

Terminal con entorno activo. Las comillas protegen los caracteres > y < en PowerShell y bash.

Qué debería ocurrir

pip descarga Django y sus dependencias (como asgiref, sqlparse, tzdata). Al final: Successfully installed Django-6.0.x ...

Cuidado aquí

Si no ves (.venv) en el prompt, detente. Instalar ahora pondría Django en el Python global y crearía problemas difíciles de diagnosticar después.

Verifica

Comprueba

El comando terminó sin errores. No cierres la terminal todavía; el siguiente paso confirma la versión.

Error común

Could not find a version that satisfies the requirement — Revisa tu conexión a internet o que escribiste bien las comillas y los símbolos de versión.

7

Verificar la versión de Django

Qué vamos a hacer

Ejecutar el comando de Django para mostrar qué versión quedó instalada.

Por qué

Instalar no garantiza que el comando responda. Esta verificación confirma que Django está en el entorno correcto y listo para crear proyectos.

Código

Terminal — (.venv) activo 
python -m django --version

Qué debería ocurrir

Salida esperada
6.0.2

El número exacto puede variar (6.0.0, 6.0.1, etc.) siempre que empiece por 6.0.

Verifica

Comprueba

La salida es 6.0.x. Si ves No module named django, el paso 6 falló o el entorno no estaba activo. Repite desde el paso 3.

Error común

django-admin is not recognized — No uses django-admin directamente todavía. Prefiere python -m django; así te aseguras de usar el Django del entorno activo.

8

Revisar paquetes con pip list

Qué vamos a hacer

Listar todos los paquetes instalados en el entorno virtual activo.

Por qué

Django no viaja solo: trae dependencias. Ver la lista completa te enseña qué hay dentro de tu "mochila" y te ayuda a detectar instalaciones accidentales en el entorno equivocado.

Código

Terminal — (.venv) activo 
python -m pip list

Qué debería ocurrir

Verás una tabla con al menos:

  • Django — versión 6.0.x
  • asgiref
  • sqlparse
  • tzdata (en Windows)
  • pip

Verifica

Comprueba

Django aparece en la lista con versión 6.0.x. Si la lista es muy corta o no incluye Django, algo falló en el paso 6.

9

Generar requirements.txt

Qué vamos a hacer

Exportar la lista exacta de paquetes y versiones a un archivo requirements.txt.

Por qué

Si compartes el proyecto o lo abres en otra computadora, requirements.txt permite recrear el mismo entorno con pip install -r requirements.txt. Es la forma estándar de documentar dependencias en Python.

Código

Terminal — (.venv) activo 
python -m pip freeze > requirements.txt

Dónde va

mi_primer_django/requirements.txt

El archivo se crea en la raíz de mi_primer_django, al mismo nivel que .venv.

Qué debería ocurrir

Aparece el archivo en el explorador de VS Code. Al abrirlo, verás líneas como:

requirements.txt 
asgiref==3.8.1
Django==6.0.2
sqlparse==0.5.3
tzdata==2025.1

Las versiones exactas pueden variar según cuándo instales.

mi_primer_django/ ├── .venv/ ├── requirements.txt └── (listo para el proyecto Django)

Verifica

Comprueba

requirements.txt existe y contiene una línea que empieza por Django==6.0.

10

Desactivar el entorno y checkpoint final

Qué vamos a hacer

Salir del entorno virtual con deactivate y confirmar que todo lo construido sigue en su sitio.

Por qué

Desactivar devuelve la terminal al Python global. Es buena práctica saber entrar y salir del entorno. La carpeta .venv no desaparece: solo dejas de usarla hasta la próxima activación.

Código

Terminal — (.venv) activo 
deactivate

Qué debería ocurrir

El prefijo (.venv) desaparece del prompt. Si ejecutas python -m django --version ahora, probablemente fallará porque Django vive dentro del entorno, no en el sistema global. Eso es correcto.

Lista de verificación final

  • Existe la carpeta mi_primer_django/.venv.
  • Puedes activar el entorno y ver (.venv) en el prompt.
  • Con el entorno activo, python -m django --version muestra 6.0.x.
  • requirements.txt existe y lista Django.
  • Sabes activar y desactivar el entorno en tu sistema operativo.

Mensaje de Teacher Jim

En el Tutorial 03 volverás a activar este mismo entorno y crearás tu primer proyecto Django. No borres mi_primer_django. Cada vez que trabajes en él, activa .venv primero.

Verifica

Comprueba

Marca este paso como hecho y pulsa "Marcar como completado" en la barra lateral. Luego continúa al Tutorial 03.

Mini reto

Practica el ciclo completo sin mirar los pasos:

  1. Desactiva el entorno si sigue activo.
  2. Vuelve a activarlo.
  3. Verifica con where python o which python que apunta a .venv.
  4. Ejecuta python -m django --version.
  5. Desactiva de nuevo.

Si puedes hacerlo en menos de dos minutos sin errores, dominas la mecánica básica del entorno virtual.
