Tutorial 05

Errores comunes de Django

Este no es un tutorial lineal: es tu centro de referencia cuando algo falla. Busca por palabra clave, lee el síntoma que reconoces y sigue el diagnóstico hasta la verificación. Cada error incluye lo que no debes hacer para no empeorar el problema.

Consulta rápida 25 errores ~30 min lectura

Cómo usar esta página

Escribe en el buscador el mensaje que ves en pantalla o una palabra clave como csrf, migrate o template. Los resultados se filtran al instante. Si no encuentras tu error, revisa primero que el entorno virtual esté activo y que estés en la carpeta correcta del proyecto; muchos fallos comparten la misma raíz.

Nota de Teacher Jim

Un error no significa que no sirvas para programar. Significa que el sistema te está dando una pista. Lee el mensaje completo, copia la última línea del traceback y búscala aquí. Resolver errores es parte del oficio.

Python no se reconoce como comando

Síntoma

Escribes python --version en la terminal y el sistema responde que no encuentra el comando.

Mensaje típico

'python' no se reconoce como un comando interno o externo...

Causa

Python no está instalado, o está instalado pero no se agregó al PATH del sistema. En Windows esto ocurre con frecuencia si no marcaste "Add python.exe to PATH" durante la instalación.

Diagnóstico

Prueba en este orden: python --version, py --version, python3 --version. Si ninguno responde, Python no está accesible desde esa terminal. Cierra y abre una terminal nueva después de instalar.

Solución

Instala Python 3.10+ desde python.org. En Windows marca "Add python.exe to PATH". Si ya instalaste sin marcarla, desinstala y reinstala, o agrega manualmente la carpeta de Python a las variables de entorno.

Verificación

python --version muestra Python 3.x.x sin errores en una terminal recién abierta.

Qué NO hacer

No copies rutas absolutas de Python en cada comando para evitar el problema. Arregla el PATH una vez y trabaja con normalidad.

No module named 'django'

Síntoma

Al ejecutar python manage.py o import django aparece un ModuleNotFoundError relacionado con django.

Mensaje típico

ModuleNotFoundError: No module named 'django'

Causa

Django no está instalado en el intérprete de Python que estás usando, o el entorno virtual no está activo y estás usando el Python global sin Django.

Diagnóstico

Verifica si ves (.venv) en el prompt. Ejecuta pip show django. Si dice "Package(s) not found", Django no está en ese entorno. Comprueba con where python (Windows) o which python (macOS/Linux) que la ruta apunte a .venv.

Solución

Activa el entorno virtual y ejecuta pip install "Django>=6.0,<6.1". Si ya estaba activo, reinstala con pip install -r requirements.txt.

Verificación

python -m django --version muestra una versión 6.0.x con el entorno activo.

Qué NO hacer

No instales Django con pip install django fuera del venv pensando que "ya quedó". Contamina el Python global y crea confusión entre proyectos.

django-admin no se reconoce

Síntoma

Escribes django-admin startproject y la terminal dice que no reconoce el comando.

Mensaje típico

'django-admin' no se reconoce como un comando interno o externo...

Causa

El entorno virtual no está activo, Django no está instalado en el entorno activo, o la carpeta Scripts/bin del venv no está en el PATH de esa sesión.

Diagnóstico

Confirma (.venv) en el prompt. Ejecuta pip show django. Prueba la forma alternativa: python -m django --version. Si esta funciona pero django-admin no, el problema es solo el script en PATH; usa la forma con python -m django.

Solución

Activa el venv, instala Django si falta, y usa python -m django startproject nombre_proyecto como alternativa robusta a django-admin.

Verificación

django-admin --version o python -m django --version responden con 6.0.x.

Qué NO hacer

No descargues django-admin por separado. Es parte del paquete Django y solo existe donde instalaste Django.

Entorno virtual no activo

Síntoma

Instalaste Django pero los comandos fallan, pip instala en otro lugar, o no ves el prefijo (.venv) en la terminal.

Mensaje típico

No hay un mensaje único; a menudo combina "no module named django" o rutas de pip que no incluyen .venv.

Causa

Creaste el entorno virtual pero no lo activaste en la terminal actual. Cada terminal nueva requiere activación manual.

Diagnóstico

Mira el inicio del prompt: debe mostrar (.venv). Ejecuta where python y confirma que la ruta contiene .venv\Scripts o .venv/bin.

Solución

PowerShell: .\.venv\Scripts\Activate.ps1. CMD: .venv\Scripts\activate.bat. macOS/Linux: source .venv/bin/activate. Luego repite el comando que falló.

Verificación

El prompt muestra (.venv) y pip --version incluye la ruta del entorno virtual.

Qué NO hacer

No asumas que abrir VS Code activa el venv automáticamente. La terminal integrada puede arrancar sin él hasta que lo actives o selecciones el intérprete correcto.

PowerShell bloquea la activación del venv

Síntoma

Al ejecutar Activate.ps1, PowerShell rechaza el script por política de ejecución.

Mensaje típico

No se puede cargar el archivo ... Activate.ps1 porque la ejecución de scripts está deshabilitada en este sistema.

Causa

La política de ejecución de PowerShell impide correr scripts locales, incluido el script de activación del entorno virtual.

Diagnóstico

Ejecuta Get-ExecutionPolicy -List. Si CurrentUser es Restricted, no podrás activar el venv con el script .ps1.

Solución

Ejecuta una vez: Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser. Confirma y vuelve a activar. Alternativa inmediata: usa CMD con .venv\Scripts\activate.bat.

Verificación

.\.venv\Scripts\Activate.ps1 funciona sin error y aparece (.venv) en el prompt.

Qué NO hacer

No uses Set-ExecutionPolicy Unrestricted sin necesidad. RemoteSigned en CurrentUser es suficiente y más seguro.

manage.py no encontrado

Síntoma

Ejecutas python manage.py runserver y Python no encuentra el archivo.

Mensaje típico

python: can't open file 'manage.py': [Errno 2] No such file or directory

Causa

La terminal no está en la carpeta del proyecto Django. manage.py vive en la raíz del proyecto creado con startproject, no en la carpeta padre ni dentro de una app.

Diagnóstico

Ejecuta dir (Windows) o ls (macOS/Linux). Debes ver manage.py en el listado. Si no aparece, sube o baja de carpeta con cd hasta encontrarlo.

Solución

Navega a la carpeta que contiene manage.py. En VS Code usa Archivo > Abrir carpeta y selecciona la raíz del proyecto. Abre una terminal nueva desde ahí.

Verificación

python manage.py --help muestra la lista de subcomandos sin error de archivo no encontrado.

Qué NO hacer

No copies manage.py a otra carpeta para "acercarlo". El archivo debe permanecer en la raíz del proyecto junto a la carpeta del paquete de configuración.

TemplateDoesNotExist

Síntoma

La vista intenta renderizar una plantilla y Django muestra una página amarilla de error indicando que no la encuentra.

Mensaje típico

TemplateDoesNotExist at /ruta/ nombre_plantilla.html

Causa

El archivo HTML no existe, el nombre en render() no coincide (mayúsculas, extensión, ruta), la app no está en INSTALLED_APPS, o la carpeta templates/ no sigue la estructura esperada.

Diagnóstico

Lee la sección "Template-loader postmortem" del error: lista las rutas donde Django buscó. Compara con la ruta real del archivo en tu proyecto. Revisa render(request, 'app/plantilla.html') carácter por carácter.

Solución

Crea la plantilla en tu_app/templates/tu_app/plantilla.html (recomendado con namespace) o ajusta el string en la vista para que coincida exactamente. Verifica que la app esté en INSTALLED_APPS.

Verificación

Recarga la URL y la página renderiza sin TemplateDoesNotExist. El HTML aparece en el navegador.

Qué NO hacer

No pongas templates sueltos en la raíz del proyecto sin configurar DIRS en TEMPLATES. Sigue la estructura de carpetas de la app.

NoReverseMatch

Síntoma

Al cargar una página o usar {% url %} en una plantilla, Django no puede construir la URL.

Mensaje típico

NoReverseMatch at / Reverse for 'nombre' not found. 'nombre' is not a valid view function or pattern name.

Causa

El nombre usado en {% url 'nombre' %} o reverse('nombre') no existe en urls.py, está mal escrito, requiere argumentos que no pasaste, o el namespace no coincide.

Diagnóstico

Busca en urls.py el name= de la ruta. Si usas app_name o include() con namespace, la referencia debe ser 'app:nombre'. Comprueba si la vista necesita pk o slug.

Solución

Corrige el nombre en la plantilla o agrega name='...' en urls.py. Si la URL necesita parámetros: {% url 'detalle' tarea.id %}. Alinea namespace y nombres entre urls del proyecto y de la app.

Verificación

La página carga sin NoReverseMatch y los enlaces generados apuntan a rutas válidas.

Qué NO hacer

No hardcodees URLs como /tareas/5/ en plantillas cuando puedes usar {% url %}. Si el nombre cambia, todo se rompe en silencio.

OperationalError: no such table

Síntoma

Al acceder a una vista que consulta la base de datos, el servidor devuelve error 500 con OperationalError.

Mensaje típico

django.db.utils.OperationalError: no such table: mi_app_modelo

Causa

Definiste un modelo en models.py pero no ejecutaste las migraciones. La tabla no existe en la base de datos SQLite.

Diagnóstico

Ejecuta python manage.py showmigrations. Si tu app muestra casillas vacías [ ] junto a migraciones pendientes, no se aplicaron. Revisa que la app esté en INSTALLED_APPS.

Solución

python manage.py makemigrations seguido de python manage.py migrate. Reinicia el servidor si estaba corriendo.

Verificación

showmigrations muestra [X] en todas las migraciones de tu app y la vista carga sin OperationalError.

Qué NO hacer

No edites db.sqlite3 a mano ni borres migraciones ya aplicadas sin saber qué haces. Eso deja el historial inconsistente.

Cambié el modelo pero la base de datos no refleja los cambios

Síntoma

Agregaste un campo al modelo, la app falla al guardar o consultar, o ves errores sobre columnas inexistentes.

Mensaje típico

OperationalError: table mi_app_modelo has no column named nuevo_campo

Causa

Modificaste models.py pero no generaste ni aplicaste una nueva migración. El código Python y el esquema de la base de datos quedaron desincronizados.

Diagnóstico

Compara models.py con la última migración en migrations/. Ejecuta python manage.py makemigrations y observa si detecta cambios.

Solución

python manage.py makemigrations para crear el archivo de migración, luego python manage.py migrate para aplicarlo. En desarrollo, revisa el archivo de migración generado antes de aplicar.

Verificación

El nuevo campo funciona en el admin o en el shell: python manage.py shell y crea un objeto con el campo nuevo.

Qué NO hacer

No reinicies el servidor esperando que Django "detecte" los cambios del modelo automáticamente. Los modelos no se sincronizan solos con la base de datos.

403 Forbidden — verificación CSRF fallida

Síntoma

Al enviar un formulario POST, Django responde con error 403 y mensaje sobre CSRF.

Mensaje típico

Forbidden (403) CSRF verification failed. Request aborted.

Causa

El formulario no incluye {% csrf_token %}, la cookie de sesión expiró, abriste el HTML como archivo local (file://), o el token no se envió en el POST.

Diagnóstico

Inspecciona el HTML del formulario en el navegador. Debe existir un input type="hidden" name="csrfmiddlewaretoken". Confirma que accedes vía http://127.0.0.1:8000, no abriendo el archivo directamente.

Solución

Dentro de <form method="post"> agrega {% csrf_token %} justo después de la etiqueta form. Recarga la página antes de enviar de nuevo para obtener un token fresco.

Verificación

El formulario POST se procesa correctamente y redirige o muestra el resultado esperado sin 403.

Qué NO hacer

No desactives CSRF con @csrf_exempt para "arreglar rápido" un formulario de login o tareas. Eso abre un agujero de seguridad real.

404 — Página no encontrada (ruta Django)

Síntoma

El servidor Django responde con una página 404 amarilla (DEBUG=True) o una página 404 genérica.

Mensaje típico

Page not found (404) Request URL: http://127.0.0.1:8000/ruta-incorrecta/

Causa

La URL que visitas no está definida en urls.py, tiene una barra de más o de menos, el orden de urlpatterns importa, o estás mezclando rutas del proyecto y de la app incorrectamente.

Diagnóstico

Lee "The current path ... didn't match any of these" en la página de debug. Compara con tus urlpatterns. Verifica includes y prefijos como path('tareas/', include('tareas.urls')).

Solución

Agrega o corrige la ruta en urls.py. Asegúrate de que el servidor se reinició si cambiaste urls (en desarrollo suele recargar solo, pero comprueba). Usa {% url %} en enlaces para evitar typos.

Verificación

La URL carga la vista correcta con código 200 en el navegador.

Qué NO hacer

No confundas el 404 de Django con el 404 de este sitio estático. Si ves la página amarilla de Django, el servidor funciona; la ruta es la que falta.

Puerto 8000 ya en uso

Síntoma

Al ejecutar runserver, Django no puede iniciar porque el puerto está ocupado.

Mensaje típico

Error: That port is already in use.

Causa

Ya tienes otro runserver corriendo en esa terminal u otra, otra aplicación usa el puerto 8000, o cerraste la terminal sin detener el proceso.

Diagnóstico

Busca otra terminal con el servidor activo. En Windows: netstat -ano | findstr :8000. En macOS/Linux: lsof -i :8000.

Solución

Detén el servidor anterior con Ctrl+C en su terminal. Si no encuentras la terminal, mata el proceso por PID o usa otro puerto: python manage.py runserver 8001.

Verificación

python manage.py runserver arranca y muestra "Starting development server at http://127.0.0.1:8000/".

Qué NO hacer

No abras múltiples runserver en el mismo puerto pensando que "el último gana". Solo uno puede escuchar en 8000.

ModuleNotFoundError (app o módulo propio)

Síntoma

Al iniciar el proyecto o importar una vista, Python no encuentra un módulo de tu proyecto.

Mensaje típico

ModuleNotFoundError: No module named 'tareas' (o 'mi_proyecto.settings')

Causa

La app no está en INSTALLED_APPS, el nombre del paquete tiene un typo, falta __init__.py (en versiones antiguas), o la estructura de carpetas no coincide con el import.

Diagnóstico

Verifica que la carpeta de la app existe junto a manage.py. Comprueba INSTALLED_APPS en settings.py. Alinea el nombre del import en urls.py con el nombre real de la carpeta.

Solución

Agrega la app a INSTALLED_APPS: 'tareas.apps.TareasConfig' o 'tareas'. Corrige typos en imports. Si creaste la app con startapp, usa el nombre exacto que generó.

Verificación

python manage.py check no reporta errores de importación y el servidor arranca.

Qué NO hacer

No renombres carpetas de apps a mano sin actualizar INSTALLED_APPS, apps.py y todos los imports.

SyntaxError

Síntoma

El servidor no arranca o una vista falla al importarse. Python señala una línea con error de sintaxis.

Mensaje típico

SyntaxError: invalid syntax (views.py, line 12)

Causa

Comillas sin cerrar, paréntesis desbalanceados, dos puntos olvidados después de if/for/def, o mezcla de comillas tipográficas copiadas desde un documento.

Diagnóstico

Lee la línea indicada y la anterior. Python a veces señala donde detectó el problema, no donde lo causaste. VS Code subraya errores de sintaxis antes de ejecutar.

Solución

Corrige la sintaxis en la línea señalada. Compara con el código del tutorial. Guarda el archivo y deja que el servidor recargue.

Verificación

El archivo importa sin SyntaxError y python manage.py check pasa.

Qué NO hacer

No comentes bloques enteros al azar para "silenciar" el error. Arregla la sintaxis; los comentarios no curan paréntesis rotos.

IndentationError

Síntoma

Python rechaza un archivo por indentación incorrecta, a menudo después de agregar código dentro de una función o clase.

Mensaje típico

IndentationError: unexpected indent (views.py, line 8)

Causa

Mezcla de tabs y espacios, bloque dentro de if/for/def sin indentar, o copiar código con indentación dañada desde la web.

Diagnóstico

En VS Code activa "Render Whitespace" para ver tabs vs espacios. Ve a la línea indicada y revisa que todos los niveles usen 4 espacios consistentemente.

Solución

Selecciona el bloque afectado y usa "Convert Indentation to Spaces". Reindenta el cuerpo de funciones y clases con 4 espacios por nivel.

Verificación

El módulo se importa sin IndentationError.

Qué NO hacer

No mezcles tabs y espacios aunque "se vean igual". Python 3 los trata como distintos y fallará.

CSS no carga — la página se ve sin estilos

Síntoma

El HTML aparece pero sin colores, fuentes ni diseño. En DevTools la hoja de estilo devuelve 404.

Mensaje típico

GET /static/css/estilos.css HTTP/1.1" 404

Causa

No usas {% load static %} y {% static 'ruta' %}, el archivo no está en la carpeta static de la app, STATIC_URL mal configurado, o en producción falta collectstatic.

Diagnóstico

Abre DevTools > Red y busca la petición al CSS. Revisa la ruta generada. Confirma que el archivo existe en app/static/app/css/... y que settings incluye django.contrib.staticfiles.

Solución

En la plantilla: {% load static %} y <link rel="stylesheet" href="{% static 'app/css/estilos.css' %}">. En desarrollo con DEBUG=True, Django sirve static automáticamente si está bien configurado.

Verificación

La petición al CSS devuelve 200 y la página muestra los estilos correctamente.

Qué NO hacer

No uses rutas absolutas como /static/... hardcodeadas sin el tag static. Si mueves archivos, la ruta debe actualizarse en la plantilla.

Imagen no carga

Síntoma

Aparece el icono de imagen rota en el navegador. La imagen no se muestra aunque el archivo existe en el proyecto.

Mensaje típico

GET /static/img/logo.png HTTP/1.1" 404 (o ruta similar)

Causa

Misma familia de problemas que CSS: ruta incorrecta en src, archivo fuera de static/, typo en el nombre, o confusión entre archivos static y media.

Diagnóstico

Inspecciona el elemento img y copia la URL que intenta cargar. Búscala en tu árbol de carpetas. Verifica mayúsculas: en Linux el nombre importa.

Solución

Coloca la imagen en app/static/app/img/logo.png y usa <img src="{% static 'app/img/logo.png' %}" alt="...">. Para subidas de usuario usa MEDIA_URL y MEDIA_ROOT, no static.

Verificación

La imagen carga con código 200 en la pestaña Red del navegador.

Qué NO hacer

No pongas imágenes de usuario subidas en la carpeta static. Static es para archivos fijos del desarrollo; media es para contenido dinámico.

DisallowedHost

Síntoma

Al acceder al sitio con una URL distinta a localhost, Django responde con error 400 DisallowedHost.

Mensaje típico

Invalid HTTP_HOST header: '192.168.1.5:8000'. You may need to add '192.168.1.5' to ALLOWED_HOSTS.

Causa

Accedes con una IP, dominio o puerto que no está listado en ALLOWED_HOSTS en settings.py. Django bloquea hosts no autorizados por seguridad.

Diagnóstico

Lee el valor exacto del host en el mensaje de error. Abre settings.py y revisa ALLOWED_HOSTS.

Solución

En desarrollo agrega el host necesario: ALLOWED_HOSTS = ['127.0.0.1', 'localhost', '192.168.1.5']. Si pruebas desde el móvil en la misma red, agrega la IP de tu PC.

Verificación

La URL carga sin DisallowedHost usando el host que agregaste.

Qué NO hacer

No uses ALLOWED_HOSTS = ['*'] en producción. En desarrollo es tentador pero enseña un mal hábito; lista solo los hosts que realmente necesitas.

SQLite database is locked

Síntoma

Operaciones de escritura fallan intermitentemente. Migraciones o guardados en el admin muestran error de base de datos bloqueada.

Mensaje típico

django.db.utils.OperationalError: database is locked

Causa

Otro proceso tiene abierto db.sqlite3 (segundo runserver, DB Browser, terminal con shell abierta en transacción), o escrituras concurrentes que SQLite no maneja bien.

Diagnóstico

Cierra programas que puedan tener la base abierta. Verifica que no hay dos servidores de desarrollo apuntando al mismo archivo de base de datos.

Solución

Cierra conexiones extra, detén runserver duplicado, cierra DB Browser. Reinicia migrate o la operación. En desarrollo evita múltiples procesos escribiendo a la vez.

Verificación

Guardar en el admin o ejecutar migrate completa sin "database is locked".

Qué NO hacer

No borres db.sqlite3 para "desbloquear" si ya tienes datos importantes. Primero cierra procesos; borrar es pérdida de datos.

Django instalado fuera del entorno virtual

Síntoma

Django funciona en una terminal pero no en otra. pip list muestra Django en un sitio y no en otro. Versiones inconsistentes entre compañeros.

Mensaje típico

No hay un mensaje único; pip show django muestra Location fuera de .venv.

Causa

Instalaste Django con pip global antes de crear el venv, o ejecutaste pip install sin activar el entorno virtual.

Diagnóstico

Con venv activo: pip show django y revisa Location. Debe estar dentro de tu proyecto/.venv. Sin venv: where pip y compara rutas.

Solución

Activa el venv e instala ahí. Si Django quedó global y causa conflicto: pip uninstall django fuera del venv (con cuidado) e instala solo dentro de .venv. Usa requirements.txt para fijar versiones.

Verificación

pip show django con venv activo muestra Location en .venv/lib o .venv\Lib.

Qué NO hacer

No instales el mismo paquete en global y en venv esperando que "use el correcto". Solo el intérprete activo decide.

VS Code usa el intérprete incorrecto

Síntoma

VS Code subraya imports de Django como error, el botón Run falla, o la terminal integrada no coincide con el venv del proyecto.

Mensaje típico

Import "django" could not be resolved Pylance (reportMissingImports)

Causa

La extensión Python apunta al Python del sistema en lugar del de .venv. Pylance analiza con un intérprete que no tiene Django instalado.

Diagnóstico

Mira la barra inferior de VS Code: debe mostrar algo como Python 3.x.x ('.venv': venv). Si muestra otra ruta, el intérprete es incorrecto.

Solución

Ctrl+Shift+P > "Python: Select Interpreter" > elige el de .venv. Abre terminal nueva para que herede el entorno. Opcional: crea .vscode/settings.json con la ruta al python del venv.

Verificación

Los imports de django dejan de marcarse como error y la terminal integrada muestra (.venv) al activarse.

Qué NO hacer

No ignores las advertencias de Pylance desactivando el análisis. Cambia el intérprete; el aviso suele ser correcto.

Los cambios de código no aparecen en el navegador

Síntoma

Editas views.py o una plantilla pero el navegador muestra la versión anterior.

Mensaje típico

No hay error; el síntoma es comportamiento viejo persistente.

Causa

No guardaste el archivo (Ctrl+S), caché del navegador, editas un archivo distinto al que usa la vista, o el servidor no recargó por un error de sintaxis silencioso en otro módulo.

Diagnóstico

Confirma que la pestaña no tiene punto sin guardar. Mira la terminal del runserver: debe decir "Watching for file changes". Haz recarga forzada: Ctrl+Shift+R. Agrega un texto único temporal en la plantilla para confirmar que es el archivo correcto.

Solución

Guarda todos los archivos. Reinicia runserver con Ctrl+C y vuelve a iniciar. Verifica que editas el proyecto correcto si tienes varias carpetas abiertas.

Verificación

El cambio visible aparece tras guardar y recargar. La terminal muestra reload al guardar.

Qué NO hacer

No asumas que el navegador siempre muestra lo último. Verifica guardado y recarga forzada antes de pensar que Django "no funciona".

El usuario ve tareas u objetos de otros usuarios

Síntoma

Al iniciar sesión, un usuario ve registros que pertenecen a otra cuenta. La lista muestra todos los objetos de la base de datos.

Mensaje típico

No hay error HTTP; es un fallo de lógica y privacidad de datos.

Causa

La vista usa Modelo.objects.all() sin filtrar por request.user. Falta ForeignKey al usuario en el modelo o no filtras en get_queryset.

Diagnóstico

Revisa la vista de listado. Busca .objects.all(). Comprueba que el modelo tiene campo usuario = models.ForeignKey(User, ...) o similar.

Solución

Filtra por usuario: Tarea.objects.filter(usuario=request.user). En vistas basadas en clases usa get_queryset() para retornar solo los del usuario autenticado. Asegura @login_required en la vista.

Verificación

Crea dos usuarios con tareas distintas. Cada uno solo ve las suyas al iniciar sesión.

Qué NO hacer

No confíes en ocultar enlaces en la plantilla. Si la URL existe y no filtras en el servidor, cualquiera puede acceder por URL directa.

El formulario POST no hace nada o falla

Síntoma

Pulsas enviar y la página se recarga sin guardar, muestra error, o los datos no llegan a la vista.

Mensaje típico

Varía: 403 CSRF, 405 Method Not Allowed, o simplemente GET con parámetros en la URL.

Causa

El form no tiene method="post", falta action correcto, la vista solo implementa GET, CSRF ausente, o el botón no es type="submit".

Diagnóstico

Inspecciona el form en HTML. Verifica method="post". En la vista confirma if request.method == 'POST':. Revisa la terminal del servidor al enviar: debe mostrar POST, no solo GET.

Solución

Formulario: <form method="post" action=""> con {% csrf_token %}. Vista: maneja POST, valida con Form o manualmente, guarda y redirige con redirect() para evitar reenvío duplicado.

Verificación

Enviar el form crea o actualiza el registro. La terminal muestra POST 302 o 200 según tu flujo. Los datos persisten tras recargar.

Qué NO hacer

No uses GET para acciones que modifican datos (crear, borrar, editar). GET debe ser idempotente y seguro de repetir.
