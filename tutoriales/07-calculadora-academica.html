Calculadora Académica

Herramienta web para calcular el promedio final a partir de actividades, porcentajes y notas. Empieza sin base de datos: formulario + lógica Python + resultado en plantilla. Ideal para una clase enfocada en validación y cálculos.

Fase 1 1 clase 10 pasos Sin DB al inicio

Resultado final

  • App grades con calculadora en /calculadora/.
  • Formulario de estudiante, materia y varias actividades (formset).
  • Promedio ponderado, estado (Excelente / Aprobado / En riesgo / Reprobado) y recomendación.
  • Tabla de aportes por actividad y CSS claro para captura de portafolio.
  • Sección bonus opcional con modelo GradeItem si quieres persistir.

Por qué sin DB primero

Te obliga a dominar formularios, validación y lógica pura. Luego añadir un modelo es opcional, no un requisito para que el proyecto funcione.

mi_primer_django/ ├── grades/ │ ├── forms.py │ ├── services.py │ ├── views.py │ ├── urls.py │ ├── models.py ← vacío al inicio; bonus al final │ └── templates/grades/ │ └── calculator.html └── static/css/calculator.css

Antes de empezar

  • Tutoria­les 01–03 listos (proyecto Django corre).
  • Tienes base.html con bloque content (del Tutorial 04 o equivalente).
  • No necesitas login para la versión básica de este proyecto.

Nota de Teacher Jim

Este proyecto cabe en una clase si no te dispersas. Primero calcula bien. El bonus con modelo es solo si te sobra tiempo.

Arquitectura

POST del formulario → validación de Form + FormSet → compute_final() en services.py → contexto con promedio/estado → mismo template muestra resultado. Sin migraciones en el camino principal.

2

Lógica pura en services.py

Qué vamos a hacer

Escribir funciones que calculan el promedio ponderado y clasifican el estado académico.

Por qué

Separar lógica de la vista facilita pruebas mentales y evita plantillas con aritmética. También es un buen talking point en portafolio.

Código

grades/services.py 
# grades/services.py — lógica pura, fácil de testear sin base de datos
def classify_average(average: float) -> tuple[str, str]:
    """Devuelve (estado, recomendación) según el promedio 0-100."""
    if average >= 90:
        return "Excelente", "Mantén el ritmo. Refuerza con un proyecto práctico."
    if average >= 70:
        return "Aprobado", "Vas bien. Repasa los temas con menor nota."
    if average >= 60:
        return "En riesgo", "Prioriza tutorías y práctica guiada esta semana."
    return "Reprobado", "Agenda un plan de recuperación y evalua qué actividades bajarion más."


def compute_final(activities: list[dict]) -> dict:
    """
    activities: [{name, grade, weight}, ...]
    grade: 0-100, weight: porcentaje (suma ideal 100).
    """
    if not activities:
        raise ValueError("Debes agregar al menos una actividad.")

    total_weight = sum(float(a["weight"]) for a in activities)
    if total_weight <= 0:
        raise ValueError("La suma de porcentajes debe ser mayor que cero.")

    weighted = 0.0
    rows = []
    for item in activities:
        grade = float(item["grade"])
        weight = float(item["weight"])
        if grade < 0 or grade > 100:
            raise ValueError(f"La nota de '{item['name']}' debe estar entre 0 y 100.")
        if weight < 0:
            raise ValueError(f"El porcentaje de '{item['name']}' no puede ser negativo.")
        contribution = grade * (weight / 100.0)
        weighted += contribution
        rows.append({
            "name": item["name"],
            "grade": grade,
            "weight": weight,
            "contribution": round(contribution, 2),
        })

    # Normaliza si la suma de pesos no es exactamente 100
    average = round(weighted * (100.0 / total_weight), 2) if total_weight != 100 else round(weighted, 2)
    status, recommendation = classify_average(average)
    return {
        "rows": rows,
        "total_weight": round(total_weight, 2),
        "average": average,
        "status": status,
        "recommendation": recommendation,
    }

Normalización

Si los porcentajes suman 80 o 120, normalizamos para no “inflar” o “congelar” el promedio de forma silenciosa. Documenta ese comportamiento en el README.

3

Formularios y formset

Qué vamos a hacer

Crear un form de metadatos (estudiante, materia) y un form de actividad para usarlo en un formset.

Por qué

El formset permite varias actividades en un solo POST sin inventar nombres de campos a mano.

Código

grades/forms.py 
from django import forms


class CalculatorMetaForm(forms.Form):
    student_name = forms.CharField(label="Nombre del estudiante", max_length=120)
    subject_name = forms.CharField(label="Materia", max_length=120)


class ActivityForm(forms.Form):
    name = forms.CharField(label="Actividad", max_length=100)
    grade = forms.FloatField(label="Nota (0-100)", min_value=0, max_value=100)
    weight = forms.FloatField(label="Porcentaje", min_value=0.01, max_value=100)

Dónde va

grades/forms.py.

Qué debería ocurrir

Importas los forms en la vista sin error.

Verifica

Comprueba

grade y weight usan min/max para validación básica de Django.

4

Vista calculator

Qué vamos a hacer

Conectar formset + meta form, llamar a compute_final y devolver resultado o error.

Por qué

La vista orquesta; no calcula. Eso mantiene el código legible.

Código

grades/views.py 
from django.forms import formset_factory
from django.shortcuts import render
from .forms import ActivityForm, CalculatorMetaForm
from .services import compute_final


ActivityFormSet = formset_factory(ActivityForm, extra=3, min_num=1, validate_min=True)


def calculator(request):
    result = None
    error = None

    if request.method == "POST":
        meta = CalculatorMetaForm(request.POST, prefix="meta")
        formset = ActivityFormSet(request.POST, prefix="acts")
        if meta.is_valid() and formset.is_valid():
            activities = []
            for form in formset:
                if not form.cleaned_data or form.cleaned_data.get("DELETE"):
                    continue
                if not form.cleaned_data.get("name"):
                    continue
                activities.append({
                    "name": form.cleaned_data["name"],
                    "grade": form.cleaned_data["grade"],
                    "weight": form.cleaned_data["weight"],
                })
            try:
                computed = compute_final(activities)
                result = {
                    **computed,
                    "student_name": meta.cleaned_data["student_name"],
                    "subject_name": meta.cleaned_data["subject_name"],
                }
            except ValueError as exc:
                error = str(exc)
        else:
            error = "Revisa los campos marcados. Todas las actividades usadas deben ser válidas."
    else:
        meta = CalculatorMetaForm(prefix="meta")
        formset = ActivityFormSet(prefix="acts")

    return render(
        request,
        "grades/calculator.html",
        {
            "meta": meta,
            "formset": formset,
            "result": result,
            "error": error,
        },
    )

Template único formulario + resultado

Qué vamos a hacer

Crear calculator.html que muestra el form y, si hay resultado, la sección de promedio.

Por qué

Un solo template simplifica el flujo de una clase. El usuario calcula y ve el resultado debajo.

Código

{% extends "base.html" %}

{% block title %}Calculadora Académica{% endblock %}

<section class="calc-hero">
  <h1>Calculadora Académica</h1>
  <p>Ingresa actividades, porcentajes y notas. Obtén promedio, estado y recomendación.</p>
</section>

{% if error %}
  <div class="calc-alert">{{ error }}</div>
{% endif %}

        <tr>
          <td>{{ row.name }}</td>
          <td>{{ row.grade }}</td>
          <td>{{ row.weight }}</td>

</section>

8

Casos de validación manual

Qué vamos a hacer

Probar entradas válidas e inválidas: nota fuera de rango, sin actividades, porcentajes raros.

Por qué

Una calculadora sin validación miente. Prefiere un error claro a un promedio absurdo.

Código

Casos de prueba 
# Caso OK: Parcial 80 (30%), Proyecto 90 (40%), Examen 70 (30%) → promedio 80
# Caso riesgo: promedio entre 60 y 69.99 → "En riesgo"
# Caso error: dejar activities vacías / nota 120 → mensaje de error
# Caso pesos: 50+50=100 → sin normalización extra relevante

Dónde va

Navegador, con runserver activo.

Qué debería ocurrir

Los casos OK muestran tabla y estado; los inválidos muestran alerta, no 500.

Verifica

Comprueba

Ningún Traceback en terminal al enviar basura intencional en el form.

Bonus: modelo GradeItem (opcional)

Si te sobra tiempo, persiste cada actividad. No es obligatorio para aprobar el mini proyecto.

grades/models.py (bonus) 
from django.conf import settings
from django.db import models


class GradeItem(models.Model):
    """Bonus opcional: persistir actividades por usuario/sesión de materia."""
    user = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name="grade_items",
    )
    student_name = models.CharField(max_length=120)
    subject_name = models.CharField(max_length=120)
    activity_name = models.CharField(max_length=100)
    grade = models.FloatField()
    weight = models.FloatField()
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        ordering = ["-created_at"]

    def __str__(self):
        return f"{self.subject_name}: {self.activity_name}"

Después: makemigrations + migrate, registra en admin, y tras calcular guarda filas con GradeItem.objects.create(...) si el usuario está autenticado. La calculadora sigue funcionando sin login.

Errores comunes

  • ManagementForm data is missing or has been tampered with — Falta management_form en el HTML.
  • Prefix mismatch — Creaste el formset con prefix en la vista pero no usas el mismo al re-renderizar errores.
  • Promedio absurdo — Estás sumando notas sin ponderar, o dividiendo mal por el total de pesos.
  • TemplateDoesNotExist — Ruta debe ser grades/templates/grades/calculator.html.
  • slugify del estado — “En riesgo” → clase status-en-riesgo; revisa CSS.
