Blog Profesional con Django Admin

Creas un blog público. El contenido se gestiona en Django Admin (no inventas un CMS). Aprendes slugs, borrador/publicado, personalización del admin y vistas que solo muestran posts publicados.

Fase 2 3 clases 12 pasos Django Admin

Resultado final

  • Modelos BlogCategory y Post con slug y estado draft/published.
  • Admin personalizado: list_display, search_fields, list_filter, prepopulated_fields, acción publicar.
  • Sitio público: listado, detalle por slug, filtro por categoría, página de autor, búsqueda.
  • Los borradores nunca aparecen en la web pública.

Por qué Admin y no otro panel

Django Admin existe para CRUD interno. Para un blog de aprendizaje es la herramienta correcta: configuras, no reinventas. Luego, si el producto crece, evalúas un panel a medida.

blog/ ├── models.py ├── admin.py ├── views.py ├── urls.py └── templates/blog/ ├── post_list.html ├── post_detail.html └── author_posts.html

Antes de empezar

  • Proyecto Django con base.html y static configurados.
  • Sabes crear superusuario (createsuperuser).
  • Idealmente completaste el Mini CRM (choices + filtros).

Nota de Teacher Jim

El error más común aquí es publicar un borrador en la web porque filtraste mal. Si status no es published, no debe existir la URL pública.

Arquitectura

Escribes contenido en /admin/. El sitio en /blog/ solo lee posts status="published". Detalle por slug, no por id: URLs legibles y estables.

2

Modelos BlogCategory y Post

Qué vamos a hacer

Definir categoría con slug y posts con excerpt, content, status y published_at.

Por qué

El slug es la URL amigable. El estado controla visibilidad. published_at documenta cuándo salió a producción.

Código

blog/models.py 
from django.conf import settings
from django.db import models
from django.utils import timezone
from django.utils.text import slugify


class BlogCategory(models.Model):
    name = models.CharField(max_length=100)
    slug = models.SlugField(unique=True)

    class Meta:
        verbose_name_plural = "blog categories"
        ordering = ["name"]

    def __str__(self):
        return self.name


class Post(models.Model):
    STATUS_CHOICES = [
        ("draft", "Borrador"),
        ("published", "Publicado"),
    ]

    title = models.CharField(max_length=180)
    slug = models.SlugField(unique=True)
    author = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name="blog_posts",
    )
    category = models.ForeignKey(
        BlogCategory,
        on_delete=models.SET_NULL,
        null=True,
        blank=True,
        related_name="posts",
    )
    excerpt = models.CharField(max_length=250)
    content = models.TextField()
    status = models.CharField(max_length=20, choices=STATUS_CHOICES, default="draft")
    published_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        ordering = ["-published_at", "-created_at"]

    def __str__(self):
        return self.title

    def publish(self):
        self.status = "published"
        if self.published_at is None:
            self.published_at = timezone.now()
        self.save(update_fields=["status", "published_at"])

4

Qué es Django Admin (y qué no es)

Qué vamos a hacer

Entender el rol pedagógico del Admin antes de personalizarlo.

Por qué

Muchos principiantes quieren “hacer un panel” desde cero. Pierden semanas. Admin cubre alta/baja/cambio de contenido interno.

Código

No hay archivo nuevo. Memoriza estas ideas:

  • Admin es para staff/superuser, no para el visitante del blog.
  • Registras modelos con @admin.register o admin.site.register.
  • Personalizas listados y formularios del admin sin tocar el sitio público.
  • El sitio público es otra capa: views + templates con filtro published.

Dónde va

Tu cabeza y el README del proyecto.

Qué debería ocurrir

Puedes explicar en una frase: “Creo posts en Admin; el blog solo muestra publicados”.

Verifica

Comprueba

Escribe esa frase en el README antes de seguir.

Nota de Teacher Jim

Si no puedes explicar qué hace list_display vs list_filter, no digas que “dominas Admin”. Domínalo con este tutorial.

5

Personalizar Admin

Qué vamos a hacer

Configurar list_display, search_fields, list_filter, prepopulated_fields y una acción para publicar.

Por qué

Estas cuatro opciones son el núcleo que preguntan en evaluaciones y demuestran que no usaste el Admin “en bruto”.

Código

blog/admin.py 
from django.contrib import admin
from django.utils import timezone
from .models import BlogCategory, Post


@admin.register(BlogCategory)
class BlogCategoryAdmin(admin.ModelAdmin):
    list_display = ("name", "slug")
    prepopulated_fields = {"slug": ("name",)}
    search_fields = ("name",)


@admin.register(Post)
class PostAdmin(admin.ModelAdmin):
    list_display = ("title", "author", "category", "status", "published_at", "created_at")
    list_filter = ("status", "category", "author")
    search_fields = ("title", "excerpt", "content")
    prepopulated_fields = {"slug": ("title",)}
    date_hierarchy = "published_at"
    ordering = ("-created_at",)
    readonly_fields = ("created_at",)

    actions = ["make_published"]

    @admin.action(description="Marcar seleccionados como publicados")
    def make_published(self, request, queryset):
        updated = 0
        for post in queryset:
            if post.status != "published":
                post.status = "published"
                if post.published_at is None:
                    post.published_at = timezone.now()
                post.save(update_fields=["status", "published_at"])
                updated += 1
        self.message_user(request, f"{updated} post(s) publicados.")

Dónde va

blog/admin.py.

Qué debería ocurrir

En el admin ves columnas útiles, buscador, filtros laterales y slug autocompletado al escribir el título.

Verifica

Comprueba

Mapa rápido

7

Vistas públicas (solo published)

Qué vamos a hacer

Listado, detalle por slug y página de autor. Todas filtran status="published".

Por qué

El Admin ve todo. El visitante no. Ese borde de seguridad editorial es el valor del proyecto.

Código

blog/views.py 
from django.db.models import Q
from django.shortcuts import get_object_or_404, render
from .models import BlogCategory, Post


def post_list(request):
    posts = Post.objects.filter(status="published").select_related("author", "category")
    category_slug = request.GET.get("category", "").strip()
    q = request.GET.get("q", "").strip()

    if category_slug:
        posts = posts.filter(category__slug=category_slug)
    if q:
        posts = posts.filter(
            Q(title__icontains=q) | Q(excerpt__icontains=q) | Q(content__icontains=q)
        )

    categories = BlogCategory.objects.all()
    return render(
        request,
        "blog/post_list.html",
        {
            "posts": posts,
            "categories": categories,
            "category_slug": category_slug,
            "q": q,
        },
    )


def post_detail(request, slug):
    post = get_object_or_404(
        Post.objects.select_related("author", "category"),
        slug=slug,
        status="published",
    )
    return render(request, "blog/post_detail.html", {"post": post})


def author_posts(request, username):
    posts = (
        Post.objects.filter(status="published", author__username=username)
        .select_related("author", "category")
    )
    return render(
        request,
        "blog/author_posts.html",
        {"posts": posts, "author_username": username},
    )

Dónde va

blog/views.py.

Qué debería ocurrir

El detalle 404 si el slug es de un borrador.

Verifica

Comprueba

get_object_or_404(..., status="published") en el detalle. No confíes solo en el listado.

URLs del blog

Por qué

Orden: rutas literales (autor/...) antes del capturador <slug:slug>.

Código

from django.urls import path
from . import views

app_name = "blog"

urlpatterns = [
    path("", views.post_list, name="post_list"),
    path("autor/<str:username>/", views.author_posts, name="author_posts"),
    path("<slug:slug>/", views.post_detail, name="post_detail"),
]

Templates públicos

Qué vamos a hacer

Por qué

Código

blog/templates/blog/post_list.html 
{% extends "base.html" %}
{% block title %}Blog{% endblock %}
{% block content %}
<section class="blog-header">
  <h1>Blog</h1>
  <p>Solo entradas publicadas. Los borradores viven en el Admin.</p>
</section>

<form method="get" class="blog-filters">
  <input type="search" name="q" value="{{ q }}" placeholder="Buscar en el blog">

11

Flujo editorial draft → published

Qué vamos a hacer

Crear un borrador, confirmar que no se ve, publicarlo (acción admin o cambio de estado), confirmar que aparece y que published_at se llenó.

Por qué

Es la historia que cuentas en el video demo del portafolio.

Código

Checklist editorial 
# 1. Admin → crear Post status=Borrador
# 2. /blog/ → no aparece
# 3. /blog/<slug>/ → 404
# 4. Admin → acción "Marcar como publicados" o status=Publicado + published_at
# 5. /blog/ → aparece
# 6. Detalle OK
# 7. Filtro por categoría y búsqueda por excerpt

Dónde va

Admin + navegador.

Qué debería ocurrir

El cambio de estado controla la visibilidad al instante.

Verifica

Comprueba

Errores comunes

  • Borradores visibles en /blog/ — Falta filter(status="published") en la vista.
  • 404 en detalle de publicado — Typo en slug o status aún en draft.
  • autor/ capturado como slug — Reordena urlpatterns.
  • prepopulated_fields no actúa — Solo funciona en el formulario del Admin (JS del admin), no en tus forms públicos.
  • IntegrityError slug — Slug duplicado; hazlo único a propósito.
  • published_at vacío en publicados — Usa la acción make_published o asigna fecha al guardar.
