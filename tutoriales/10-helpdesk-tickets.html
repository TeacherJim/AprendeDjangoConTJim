Proyecto profesional: HelpDesk de Tickets

Vas a construir un sistema de soporte real: el usuario crea tickets; el staff ve todos y responde. Aprendes prioridades, estados, filtros y la diferencia entre usuario normal y is_staff.

Proyecto profesional Fase 3 4 clases 11 pasos

Resultado final

Al terminar tendrás una aplicación Django con:

  • Modelo Ticket con prioridad, estado, categoría y respuesta del staff.
  • Usuarios autenticados que crean tickets y solo ven los suyos.
  • Cuentas is_staff=True que ven todos los tickets y pueden cambiar estado y responder.
  • Lista filtrable por estado, detalle y dashboard con contadores.
  • Formularios POST con CSRF y vistas protegidas con login_required.

Idea clave

Un HelpDesk es un CRUD con roles. No necesitamos permisos avanzados todavía: request.user.is_staff basta para ensearte el patrón.

Antes de empezar

  • Completaste autenticación (Tutorial 04) y al menos un CRUD con usuario (Notes o CRM).
  • Sabes crear un proyecto Django nuevo o reutilizar uno con login ya funcionando.
  • Entorno virtual activo y Django instalado.
  • Puedes crear un superusuario con createsuperuser.

Nota de Teacher Jim

Este proyecto brilla en el portafolio porque demuestra roles. En la demo: muestra un usuario normal creando un ticket y luego entra como staff y responde. Ese contraste vale más que mil capturas sin contexto.

2

Modelo Ticket con prioridades y estados

Qué vamos a hacer

Definir el modelo Ticket con PRIORITY_CHOICES, STATUS_CHOICES y campos de respuesta del staff.

Por qué

Los choices evitan valores inventados en la base de datos y muestran etiquetas legibles en formularios y plantillas.

Código

tickets/models.py 
from django.conf import settings
from django.db import models


class Ticket(models.Model):
    PRIORITY_CHOICES = [
        ("low", "Baja"),
        ("medium", "Media"),
        ("high", "Alta"),
    ]

    STATUS_CHOICES = [
        ("open", "Abierto"),
        ("in_progress", "En proceso"),
        ("resolved", "Resuelto"),
        ("closed", "Cerrado"),
    ]

    user = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name="tickets",
    )
    title = models.CharField(max_length=180)
    description = models.TextField()
    category = models.CharField(max_length=100)
    priority = models.CharField(
        max_length=20,
        choices=PRIORITY_CHOICES,
        default="medium",
    )
    status = models.CharField(
        max_length=20,
        choices=STATUS_CHOICES,
        default="open",
    )
    staff_response = models.TextField(blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ["-created_at"]

    def __str__(self):
        return f"#{self.pk} {self.title}"

Dónde va

Reemplaza el contenido de tickets/models.py.

Qué debería ocurrir

El archivo importa sin errores de sintaxis. Aún no hay tablas: faltan migraciones.

Nota de Teacher Jim

Usamos settings.AUTH_USER_MODEL en lugar de importar User directo. Es la forma correcta desde el día uno: si algún día cambias el modelo de usuario, las ForeignKey no se rompen.

5

Formularios de usuario y de staff

Qué vamos a hacer

Crear TicketForm (usuario) y StaffTicketForm (estado + respuesta).

Por qué

El usuario no debe poder ponerse un ticket en resolved solo. El staff sí puede cambiar estado y escribir staff_response.

Código

tickets/forms.py 
from django import forms
from .models import Ticket


class TicketForm(forms.ModelForm):
    class Meta:
        model = Ticket
        fields = ["title", "description", "category", "priority"]
        widgets = {
            "title": forms.TextInput(attrs={"class": "form-control"}),
            "description": forms.Textarea(attrs={"class": "form-control", "rows": 5}),
            "category": forms.TextInput(attrs={"class": "form-control", "placeholder": "Ej: Acceso, Bug, Cuenta"}),
            "priority": forms.Select(attrs={"class": "form-control"}),
        }


class StaffTicketForm(forms.ModelForm):
    class Meta:
        model = Ticket
        fields = ["status", "staff_response"]
        widgets = {
            "status": forms.Select(attrs={"class": "form-control"}),
            "staff_response": forms.Textarea(attrs={"class": "form-control", "rows": 4}),
        }

Separación de responsabilidades

Dos formularios = dos permisos visuales. Aunque alguien manipule el HTML, en las vistas solo aplicarás el formulario que corresponde al rol.

6

Vistas: listar, crear, detalle y regla is_staff

Qué vamos a hacer

Implementar las vistas con filtro por usuario o todos si es staff. Incluir respuesta del staff en el detalle.

Por qué

Esta es la regla de oro del HelpDesk. Sin ella, cualquier usuario vería tickets ajenos.

Código

tickets/views.py 
from django.contrib.auth.decorators import login_required
from django.db.models import Count, Q
from django.shortcuts import get_object_or_404, redirect, render
from django.contrib import messages

from .forms import StaffTicketForm, TicketForm
from .models import Ticket


def _tickets_for(user):
    """Staff ve todos; usuario normal solo los suyos."""
    if user.is_staff:
        return Ticket.objects.select_related("user").all()
    return Ticket.objects.filter(user=user)


@login_required
def ticket_list(request):
    qs = _tickets_for(request.user)
    status = request.GET.get("status", "")
    if status:
        qs = qs.filter(status=status)
    q = request.GET.get("q", "").strip()
    if q:
        qs = qs.filter(Q(title__icontains=q) | Q(description__icontains=q) | Q(category__icontains=q))
    return render(request, "tickets/ticket_list.html", {
        "tickets": qs,
        "status": status,
        "q": q,
        "status_choices": Ticket.STATUS_CHOICES,
        "is_staff_view": request.user.is_staff,
    })


@login_required
def ticket_create(request):
    if request.method == "POST":
        form = TicketForm(request.POST)
        if form.is_valid():
            ticket = form.save(commit=False)
            ticket.user = request.user
            ticket.save()
            messages.success(request, "Ticket creado correctamente.")
            return redirect("tickets:detail", pk=ticket.pk)
    else:
        form = TicketForm()
    return render(request, "tickets/ticket_form.html", {"form": form})


@login_required
def ticket_detail(request, pk):
    base = _tickets_for(request.user)
    ticket = get_object_or_404(base, pk=pk)
    staff_form = None

    if request.user.is_staff:
        if request.method == "POST":
            staff_form = StaffTicketForm(request.POST, instance=ticket)
            if staff_form.is_valid():
                staff_form.save()
                messages.success(request, "Ticket actualizado por el equipo de soporte.")
                return redirect("tickets:detail", pk=ticket.pk)
        else:
            staff_form = StaffTicketForm(instance=ticket)

    return render(request, "tickets/ticket_detail.html", {
        "ticket": ticket,
        "staff_form": staff_form,
    })


@login_required
def dashboard(request):
    qs = _tickets_for(request.user)
    counts = qs.aggregate(
        total=Count("id"),
        open=Count("id", filter=Q(status="open")),
        in_progress=Count("id", filter=Q(status="in_progress")),
        resolved=Count("id", filter=Q(status="resolved")),
        closed=Count("id", filter=Q(status="closed")),
        high=Count("id", filter=Q(priority="high")),
    )
    recent = qs[:8]
    return render(request, "tickets/dashboard.html", {
        "counts": counts,
        "recent": recent,
        "is_staff_view": request.user.is_staff,
    })

Nota de Teacher Jim

Esta regla con is_staff es una introducción. En producción usarías grupos, permisos custom o un paquete como django-guardian. Por ahora: entiende el patrón, no inventes complejidad.

Error común

Usuario ve tickets de otros — Olvidaste filtrar con _tickets_for en ticket_detail. Siempre pasa el queryset filtrado a get_object_or_404, no Ticket.objects.all().

Entender el dashboard de contadores

Qué vamos a hacer

Repasar la agregación del dashboard: totales por estado y prioridad alta.

Por qué

Un panel con números comunica el estado del soporte mejor que una lista infinita.

Código clave

Agregación (ya en views.py) 
counts = qs.aggregate(
    total=Count("id"),
    open=Count("id", filter=Q(status="open")),
    in_progress=Count("id", filter=Q(status="in_progress")),
    resolved=Count("id", filter=Q(status="resolved")),
    closed=Count("id", filter=Q(status="closed")),
    high=Count("id", filter=Q(priority="high")),
)

Staff ve métricas globales; usuario normal ve solo las suyas, porque qs ya viene filtrado.

Comprueba

Crea 3 tickets con estados distintos y verifica que los números del dashboard coinciden.

Crear usuario normal, probar roles y verificar

Qué vamos a hacer

Arrancar el servidor, crear un segundo usuario (no staff) y validar el aislamiento.

Código

Terminal 
python manage.py check
python manage.py runserver

En el admin, crea un usuario cliente1 sin marcar Staff. Con el superusuario abre tickets; cierra sesión e inicia con cliente1.

Comprueba

cliente1 solo ve sus tickets. El superusuario ve todos y puede guardar staff_response.

Checklist final

  • Modelo Ticket con PRIORITY_CHOICES y STATUS_CHOICES.
  • Usuario crea ticket con título, descripción, categoría y prioridad.
  • Usuario normal solo ve sus tickets en lista y detalle.
  • Staff ve todos, filtra por estado y responde.
  • Dashboard muestra totales coherentes con la vista del rol.
  • Logout por POST con CSRF.
  • Repositorio limpio, sin .venv ni secretos en Git.

Errores comunes

Error común

403 Forbidden en logout — El logout debe ser POST con {% csrf_token %}, no un enlace GET.

Error común

AttributeError: get_priority_display — El método existe solo si el campo tiene choices. Revisa el nombre del campo.

Error común

Staff ve poco / usuario ve de más — Revisa _tickets_for en lista Y detalle. El fallo suele estar en el detalle.

+

Mini retos

Reto A — Comentarios internos

Crea un modelo TicketComment ligado al ticket y muestra el historial en el detalle.

Pista: ForeignKey a Ticket + ForeignKey a User; solo staff puede crear comentarios internos.

Reto B — SLA visual

Resalta en rojo tickets abiertos con más de 48 horas.

Pista: Compara timezone.now() - ticket.created_at en la vista o con un método del modelo.

Reto C — Adjuntar archivo

Añade FileField opcional al ticket y configura MEDIA_URL.
