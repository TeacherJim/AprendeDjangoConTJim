Qué debe tener un README de proyecto

Qué vamos a hacer

Lista las secciones obligatorias antes de copiar la plantilla.

  • Nombre y descripción en una frase.
  • Problema que resuelve.
  • Funcionalidades.
  • Tecnologías.
  • Capturas.
  • Instalación local paso a paso.
  • Aviso de secretos.
  • Aprendizajes y mejoras futuras.
  • Enlace a tu portafolio.

README ≠ ensayo

Frases cortas, comandos copiables, capturas reales. Quien lee quiere correr el proyecto en 5 minutos.

2

Plantilla completa de README (Markdown)

Código

Copia esto en README.md de tu repo (ejemplo HelpDesk). Adáptalo a Notes, CRM o Dashboard cambiando nombres.

README.md 
# HelpDesk de Tickets (Django)

Sistema de tickets de soporte con roles de usuario y staff, prioridades, estados y dashboard de contadores.

## Problema que resuelve

Centraliza solicitudes de ayuda: el usuario reporta un incidente y el equipo de soporte hace seguimiento hasta resolverlo.

## Funcionalidades

- Registro / login (reutiliza auth de Django)
- Crear tickets (título, descripción, categoría, prioridad)
- Usuario solo ve sus tickets
- Staff ve todos, cambia estado y responde
- Filtros por estado y búsqueda de texto
- Dashboard con totales por estado y prioridad alta

## Tecnologías

- Python 3.x
- Django 5/6
- SQLite (desarrollo)
- HTML / CSS

## Capturas


![Lista de tickets](docs/lista.png)
![Dashboard](docs/dashboard.png)

## Cómo ejecutarlo en local

```bash
git clone https://github.com/TU-USUARIO/helpdesk-django.git
cd helpdesk-django
python -m venv .venv

# Windows
.venv\Scripts\activate
# macOS / Linux
source .venv/bin/activate

pip install -r requirements.txt
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

Abre http://127.0.0.1:8000/ e inicia sesión.

## Variables de entorno / secretos

No subas `SECRET_KEY` real ni archivos `.env` con credenciales.
En desarrollo puedes usar la key de ejemplo solo en máquinas locales.

## Estructura

```text
config/          # settings y urls del proyecto
tickets/         # app principal
static/          # CSS
templates/       # login
```

## Decisiones de diseño

- Roles con `user.is_staff` (introducción; no RBAC avanzado)
- Dos ModelForm: uno para usuarios, otro para staff
- Queryset filtrado en lista y detalle para evitar fugas de datos

## Aprendizajes

- Aislar datos por usuario
- Agregar métricas con `Count` y `Q`
- Comunicar estados con badges y dashboard

## Mejoras futuras

- Adjuntos
- Notificaciones por email
- Grupos de agentes
- API REST

## Autor

Tu Nombre — [Portafolio](https://TU-USUARIO.github.io/TU-REPO-PORTAFOLIO/)

Añade también un requirements.txt congelado:

Terminal 
pip freeze > requirements.txt
3

Guion del video demo (2–3 minutos)

Qué vamos a hacer

Escribir el guion antes de grabar. Improvisar alarga el video y se pierde el mensaje.

guion-demo.txt 
Guion demo 2–3 minutos (HelpDesk — adapta a tu proyecto estrella)

[0:00–0:15] Presentación
«Hola, soy [Nombre]. Te muestro mi HelpDesk en Django: tickets con prioridad, estados y roles usuario/staff.»

[0:15–0:45] Usuario normal
- Inicio sesión como cliente1
- Creo un ticket de prioridad alta
- Muestro que en la lista solo aparecen MIS tickets

[0:45–1:30] Staff
- Cierro sesión / entro como staff
- Abro el dashboard: contadores
- Abro el ticket del cliente, cambio a «En proceso» y escribo respuesta
- Guardo y muestro la respuesta visible en el detalle

[1:30–2:15] Detalles técnicos (breve)
- «Usé ForeignKey a User, choices de prioridad/estado, y filtro con is_staff.»
- «El detalle siempre consulta el queryset filtrado para no filtrar datos ajenos.»

[2:15–2:45] Portafolio
- Abro mi GitHub Pages
- Señalo la card del proyecto y el README del repo

[2:45–3:00] Cierre
- «Siguiente paso: [permisos avanzados / email / adjuntos]. Gracias.»

  5. Súbelo a YouTube (no listado) o a un release de GitHub / Drive público.

5

Limpieza de secretos y repos

Qué vamos a hacer

Auditar que nada sensible quedó en Git.

.gitignore (mínimo Django) 
.venv/
__pycache__/
*.py[cod]
db.sqlite3
.env
*.log
staticfiles/
media/
.DS_Store
Thumbs.db
  • No hay SECRET_KEY de producción en el repo.
  • No hay contraseñas en capturas ni en el video.
  • .venv/ y db.sqlite3 están ignorados.
  • El README no pide clonar con tokens de acceso.

Error común

Si ya subiste un secreto, cámbialo (rota la clave). Borrar el commit no siempre basta si alguien clonó.

6

Checklist de graduación

Qué vamos a hacer

Marca cada ítem. Si alguno falla, ese es tu siguiente trabajo — no improvises el cierre.

  • GitHub limpio en cada proyecto Django (README + .gitignore + requirements).
  • Capturas actualizadas en docs/ o en el portafolio.
  • Portafolio publicado en GitHub Pages con URL que abre sin 404 de CSS.
  • Al menos un video demo enlazado.
  • Ningún secreto en repos públicos.
  • Cards de proyectos coherentes (problema, tech, repo, aprendizaje).
  • Puedes explicar en voz alta el flujo de tu mejor app en 3 minutos.
  • Fecha de entrega del portafolio anotada: 19 de noviembre de 2026 a las 5:00 p. m. (el 1 de diciembre es la graduación, no la entrega).

Comprueba

Pídele a un compañero que abra solo tu URL de Pages y te diga qué proyecto entendería primero. Si no entiende ninguno, reescribe las descripciones.

Errores comunes

Error común

README de una línea — «Proyecto Django». Insuficiente. Usa la plantilla del paso 2.

Error común

Video de 12 minutos sin guion — Nadie lo termina. Corta a 3 minutos con el guion del paso 3.

Error común

Subir .env o SECRET_KEY — Rota la clave y limpia el historial o empieza un repo nuevo si hace falta.
